# Postgres Migration Spike Plan (SCRUM-14)

## Objective

Deliver an implementation-ready plan for moving the current SQLite persistence model to Postgres, with local rehearsal tooling and rollback-safe operational guidance.

## Current SQLite -> Target Postgres Mapping

| SQLite table | Postgres table | Key notes |
| --- | --- | --- |
| `role_types` | `role_types` | Preserve numeric IDs and unique `name`. |
| `order_status_types` | `order_status_types` | Preserve numeric IDs and unique `name`. |
| `users` | `users` | Preserve IDs and role metadata (`role`, `role_id`, profile fields). |
| `catalog_items` | `catalog_items` | Preserve IDs, SKU, public IDs, pricing. |
| `orders` | `orders` | Preserve IDs, status mapping, payment/shipping fields; convert `created_at` to `TIMESTAMPTZ`. |
| `order_items` | `order_items` | Preserve IDs and foreign key relationships to `orders` and `catalog_items`. |

## Target Schema Decisions

- IDs use `GENERATED BY DEFAULT AS IDENTITY` to support both explicit insert IDs during migration and automatic IDs for future runtime writes.
- `orders.created_at` is `TIMESTAMPTZ` for reliable ordering/query behavior under multi-node deployment.
- Key indexes from SQLite runtime are preserved:
  - `idx_users_role_id`
  - `idx_catalog_items_public_id`
  - `idx_orders_public_order_id`
  - `idx_orders_status_type_id`
  - `idx_orders_user_created_at`

## Rehearsal Sequence

1. Bring up local Postgres (Docker Compose).
2. Apply Postgres schema.
3. Run SQLite -> Postgres rehearsal loader.
4. Run parity checks:
   - row counts across core tables,
   - relationship integrity checks on critical foreign keys.
5. Capture baseline query timings for key order-read paths.

## Rollout Plan (Production-Oriented)

1. **Pre-cutover**
   - Freeze schema changes for migration window.
   - Capture verified backup snapshot of source DB.
   - Run full rehearsal in staging with representative data volume.
2. **Cutover execution**
   - Enable write freeze at the application boundary.
   - Execute migration loader against production target.
   - Run parity checks and smoke tests before opening write traffic.
3. **Post-cutover validation**
   - Monitor key query latency and error rates.
   - Validate order creation and order history read flows.

## Abort and Rollback Plan

Abort immediately if any of the following occur:

- parity checks fail for core tables,
- critical foreign-key integrity violations are detected,
- post-cutover smoke tests fail on checkout/order history flows.

Rollback actions:

1. Keep or restore traffic to SQLite write path.
2. Restore source snapshot if freeze window was violated or corruption is suspected.
3. Open incident follow-up with mismatch report and root-cause remediation before retry.

## Out of Scope for SCRUM-14

- Production runtime database switch.
- ORM/adaptor refactor for Postgres-backed request path.
- Broad application feature changes unrelated to persistence migration readiness.
